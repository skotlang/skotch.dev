## Workspace layout

Skotch is organized as a Cargo workspace of ~25 crates arranged in a
strict dependency DAG. Lower layers know nothing about higher ones.

```
Layer 0 — primitives:    span, intern, config, diagnostics
Layer 1 — front-end:     syntax, lexer, parser
Layer 2 — semantic:      resolve, types, typeck
Layer 3 — IRs:           hir, mir, mir-lower
Layer 4 — backends:      backend-jvm, backend-dex, backend-llvm,
                         backend-klib, backend-wasm
Layer 5 — normalizers:   classfile-norm, dex-norm, llvm-norm
Layer 6 — orchestration: driver, buildscript, build
Layer 7 — packaging:     jar, apk, axml, sign
Layer 8 — runtime:       jvm (embedded JNI), repl
Layer 9 — CLI:           cli (the `skotch` binary)
```

## Design principles

### Single binary, no external tools

The shipping `skotch` binary never invokes `kotlinc`, `javac`, `d8`, or
Gradle. The only external tool it calls is `clang`, for the native
target's link step. Reference outputs used for testing are generated by
a separate `xtask` binary (the only crate allowed to shell out to
external compilers) and committed to git.

### Hand-rolled bytecode writers

JVM class files and DEX files are written directly using `byteorder`.
Constant-pool forward references make higher-level serialization
frameworks awkward for these formats, so the writers manage pool indices
and backpatches manually.

### Textual LLVM IR

The LLVM backend emits plain-text `.ll` files via string formatting.
This avoids linking against `libLLVM` (which adds a system requirement
and significant build time) at the cost of not being able to run LLVM
optimization passes in-process.

### MIR as the waist

All five backends consume the same MIR (Mid-level IR). Adding a new
target means writing one backend crate that lowers MIR to the target
format. The front-end, resolver, type checker, and MIR lowering are
shared.

### Fixture-driven validation

Test fixtures live under `tests/fixtures/inputs/`. Each fixture is
compiled by Skotch and by the reference tool for that target. Both
outputs are committed under `tests/fixtures/expected/` so CI can diff
them without installing the JDK, Android SDK, or `kotlinc-native`.

Normalizers (`classfile-norm`, `dex-norm`, `llvm-norm`) strip cosmetic
differences before comparison.

## Parallelism

Skotch uses Rayon for nested work-stealing parallelism. In a project
build, modules are compiled in parallel (respecting dependency order),
files within a module are compiled in parallel, and within each file
the MIR lowering of independent functions can be parallelized.

## Crate responsibilities

| Crate | Role |
|-------|------|
| `skotch-lexer` | Hand-rolled lexer; emits tokens including string template sequences |
| `skotch-parser` | Recursive-descent parser; produces an untyped AST |
| `skotch-resolve` | Name resolution; binds identifiers to declarations |
| `skotch-typeck` | Two-pass type checker (signatures then bodies) |
| `skotch-mir` | MIR data structures (three-address-code, SSA-like locals) |
| `skotch-mir-lower` | Lowers typed AST to MIR |
| `skotch-backend-jvm` | MIR → `.class` (Java 17) |
| `skotch-backend-dex` | MIR → `.dex` (Dalvik v035) |
| `skotch-backend-klib` | MIR → `.klib` (ZIP + JSON IR) |
| `skotch-backend-llvm` | MIR → `.ll` (textual LLVM IR) |
| `skotch-driver` | Wires front-end to backends; `emit` entry point |
| `skotch-buildscript` | Token-walker parser for `build.gradle.kts` / `settings.gradle.kts` |
| `skotch-build` | Project-level build orchestrator (source discovery, multi-module, packaging) |
| `skotch-jar` | JAR writer (ZIP with `META-INF/MANIFEST.MF`) |
| `skotch-apk` | APK assembler (ZIP with alignment, signing block insertion) |
| `skotch-axml` | Android binary XML writer (for `AndroidManifest.xml`) |
| `skotch-sign` | APK Signature Scheme v2 (RSA-PKCS1-v1.5 + SHA-256) |
| `skotch-jvm` | Embedded JVM via JNI for the REPL and script runner |
| `skotch-repl` | Interactive REPL and `.kts` script runner |
| `skotch-cli` | Binary entry point; clap-based subcommand dispatch |
